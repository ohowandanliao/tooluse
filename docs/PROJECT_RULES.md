# 项目规范

这份文档记录的是工程和文档维护细则。

用户规则入口已经单独放到根目录：

- `RULES.md`

优先级约定：

- 如果 `RULES.md`、`README.md`、`STATUS.md`、`HANDOFF.md`、实验记录或本文件之间出现表述不一致，以 `RULES.md` 为准。
- 本文件负责把 `RULES.md` 中的原则，落到工程结构、文档放置和维护动作上。

## 对齐到 `RULES.md` 的七条原则

### 原则一：尽量不自己造轮子，但不是完全不自己造轮子

这条原则的实际含义是：

- baseline、训练 runtime、数据导出、评测、依赖安装，优先复用成熟开源方案。
- 只有在下面几种情况下，才应该自己实现：
  - 研究核心假设本身无法用现成框架表达；
  - 现成工具无法稳定支持当前目标；
  - 只差一层很薄的 glue code，就能把成熟工具接进来。
- 一旦成熟工具路径已经跑通，自写替代品就不再是主线。

当前落地到本仓库的结论：

- baseline runtime 以 `LLaMA-Factory` 为主，不再扩展手写 baseline trainer。
- 旧的手写 baseline trainer 已经从代码库中移除。
- 不要重新引入只会产出 smoke stub 的自写 baseline 训练路径。

### 原则二：必须随时更新 `STATUS.md` 和 `HANDOFF.md`

这条原则不是“有空再补文档”，而是工程流程的一部分。

出现下面任一变化时，同一轮工作里就应该同步更新：

- 运行环境路径变化；
- 第三方 runtime 路径变化；
- 训练、评测、数据导出结果变化；
- 下一步建议变化；
- 已知风险、删除门槛、工程判断变化；
- 重要实验新增了可复现结论。

更新分工：

- `STATUS.md`
  - 面向“当前仓库真实状态”。
  - 记录现在能跑什么、不能证明什么、下一步最该做什么。
- `HANDOFF.md`
  - 面向“下一位接手的人”。
  - 记录接手顺序、最短进入路径、容易踩坑的地方和本机验证结果。
- `docs/records/YYYY-MM-DD-*.md`
  - 面向“单次重要运行或阶段性记录”。
  - 记录具体一次实验/打通的上下文、命令、结果和解释。

### 原则三：把仓库当作未来要 release 的开源项目维护

这意味着：

- 仓库根目录只保留源码、配置、必要文档和轻量数据接口。
- conda 环境、editable 第三方 checkout、checkpoint、预测输出、loss 图等重产物都放到仓库外。
- 文档不能只写给当前操作者看，也要写给未来的开源用户和接手开发者看。
- 目录命名、文件命名、运行命令和路径约束都要可解释、可复现、可清理。

当前本机约定：

- Python 环境：`/root/miniconda3/envs/tooluse-llf`
- 重产物根目录：`/root/autodl-fs/tooluse-artifacts`

### 原则四：不确定的东西先查外部做法，再决定是否推进

这条原则的重点不是“多看点资料”，而是避免把未经验证的工程判断直接升级成项目主线。

下面几类情况，默认先查外部论文、官方文档、主流开源工程或模型卡，再决定：

- 某个 benchmark 是否适合直接当训练集；
- 某种数据格式或评测口径是否是社区常见做法；
- 某条 runtime、prompt、微调配方是否只是某个项目的局部技巧；
- 某个看起来可行的 shortcut 是否会引入 benchmark contamination、split leakage 或不公平比较。

执行要求：

- 优先看官方 benchmark 文档、原论文、官方仓库、主流模型/项目的训练说明。
- 如果外部做法还没有形成共识，就不要把当前仓库里的探索路径包装成“默认主线”。
- 代码可以先做成可选脚手架，但 `STATUS.md`、`HANDOFF.md` 和运行说明里必须明确它的定位是：
  - 已验证主线；
  - 探索性路径；
  - 仅用于评测；
  - 或待确认。

### 原则五：仓库内要保留可复现实验证据，但不提交权重

这条原则的重点是把“实验确实跑过”的证明材料保留在 release 仓库里，同时避免仓库变成权重和缓存堆场。

当前落地要求：

- 重要实验至少保留运行命令、标量结果、精确评测结果和必要日志。
- 这些证据统一进入 `results/`。
- 权重、checkpoint、模型缓存、editable 第三方源码 checkout 仍然放仓库外。
- 如果原始预测输出很小且确实有证明价值，可以一并进入 `results/`。

### 原则六：如果需要，可以整理 prompt 让用户去问 `GPT-5.4-pro` 做扩展或纠偏

这条原则的作用，是在仓库内研究推进遇到高不确定性或需要外部强审阅时，允许显式引入一次高质量外部思考，但不把它误用成“跳过工程验证”的捷径。

当前落地要求：

- 只有在确实需要额外方案空间、外部 challenge 或更强总结能力时，才建议这样做。
- prompt 必须写清楚当前仓库上下文、已验证事实、待决问题和输出要求，避免问成泛泛而谈的大模型闲聊。
- 采纳外部回答前，仍然要回到当前仓库核查代码、实验和文档。
- 如果这条路径明显影响后续决策，应该把 prompt 用途和采纳结论写回 `STATUS.md` 或 `docs/records/`。

### 原则七：做外部检索时，尽量优先更新的资料；旧资料要么够新，要么够硬

这条原则的目标，是避免研究判断长期建立在过旧或弱影响力材料上。

当前落地要求：

- 搜论文、工程、benchmark 用法、训练 recipe 时，默认先看 `2025` 年及以后。
- 如果必须引用更早的资料，优先选择源头工作、官方文档、官方仓库或高影响力论文。
- 较旧但影响力弱、已经明显过时的材料，不应成为主判断依据。
- 如果保留了较旧材料，最好在记录里解释它为什么仍值得参考。

## 文档结构

文档按职责分层，不再把不同用途的内容混在一起。

### 根目录文档

- `README.md`
  - 面向仓库访问者和潜在开源用户。
  - 说明项目目标、快速开始、仓库结构和产物策略。
- `STATUS.md`
  - 当前真实状态。
- `HANDOFF.md`
  - 跨机器、跨会话交接。
- `results/`
  - release 仓库内的实验结果证据层，不存权重。
- `2026-03-31-nips2026-function-calling-idea-draft-v2.md`
  - 当前论文主草稿。

### `docs/` 下的文档

- `docs/PROJECT_RULES.md`
  - 本文件，记录原则和维护规范。
- `docs/llamafactory-baseline.md`
  - baseline runtime 与评测使用说明。
- `docs/environment-repro.md`
  - 新机器恢复与环境复现说明。
- `docs/records/`
  - 日期化的实验记录、bring-up 记录、资源估计、研究判断。

## 文档维护规则

### 写新文档前先判断放哪

- 如果它是仓库入口、项目总状态或交接说明，放根目录。
- 如果它是需要提交的轻量实验结果证据，放 `results/`。
- 如果它是一次实验、一次 bring-up、一次审计，放 `docs/records/`。
- 如果它只是临时执行计划、Prompt 资产或内部协作草稿，默认不要保留到 release 仓库。

### 不要再出现的情况

- 在根目录堆临时 scratch note。
- 把实验结论只写在终端输出里，不写回文档。
- 同一条关键原则在多个地方重复但长期不同步。
- 把本机重路径硬编码进多个文档却不统一维护。
- 把权重或明显重产物塞进 `results/`。
- 把只对内部协作或单次 AI 会话有意义的计划文档长期留在仓库里。

### 建议的写法

- 运行记录优先写清楚：目的、非目标、命令、结果、解释、下一步。
- 判断性内容要和证据分开写，避免把工程打通说成论文结论。
- 面向开源发布的入口文档，避免只对内部上下文可读。

## 工程与文档之间的关系

文档不是附属品，而是 runtime、实验和论文推进的一部分。

- 新的运行路径打通后，要把命令和路径写回文档。
- 路径迁移后，要把旧路径引用清掉。
- 如果新增了比旧路径更成熟的方案，文档必须明确“谁是主线，谁是 legacy”。
- 如果代码还保留 legacy 路径，文档必须说明保留原因和删除条件。
- 如果历史计划文档仍提到已删除路径，以当前代码、`README.md`、`STATUS.md` 为准。

## 当前固定阅读路径

新接手的人默认按这个顺序读：

1. `README.md`
2. `RULES.md`
3. `docs/PROJECT_RULES.md`
4. `STATUS.md`
5. `HANDOFF.md`
6. 论文主草稿
7. 相关 runtime/实验记录

补充约定：

- 根目录 `STATUS.md` 只表示“当前总状态总览”。
- `docs/records/` 只表示“阶段记录、实验记录、资源记录”。
- 以后不再新建 `docs/status/` 这种容易和 `STATUS.md` 撞名的目录。
