# 项目规范

这份文档是当前仓库的单点规范来源。

如果 `README.md`、`STATUS.md`、`HANDOFF.md`、实验记录或后续新增文档之间出现表述不一致，以本文件的原则和维护约束为准，再回头修正文档分歧。

## 三条原则

### 原则一：尽量不自己造轮子，但不是完全不自己造轮子

这条原则的实际含义是：

- baseline、训练 runtime、数据导出、评测、依赖安装，优先复用成熟开源方案。
- 只有在下面几种情况下，才应该自己实现：
  - 研究核心假设本身无法用现成框架表达；
  - 现成工具无法稳定支持当前目标；
  - 只差一层很薄的 glue code，就能把成熟工具接进来。
- 一旦成熟工具路径已经跑通，自写替代品就不再是主线。

当前落地到本仓库的结论：

- baseline runtime 以 `LLaMA-Factory` 为主，不再扩展手写 baseline trainer。
- 旧的手写 baseline trainer 已经从代码库中移除。
- 不要重新引入只会产出 smoke stub 的自写 baseline 训练路径。

### 原则二：必须随时更新 `STATUS.md` 和 `HANDOFF.md`

这条原则不是“有空再补文档”，而是工程流程的一部分。

出现下面任一变化时，同一轮工作里就应该同步更新：

- 运行环境路径变化；
- 第三方 runtime 路径变化；
- 训练、评测、数据导出结果变化；
- 下一步建议变化；
- 已知风险、删除门槛、工程判断变化；
- 重要实验新增了可复现结论。

更新分工：

- `STATUS.md`
  - 面向“当前仓库真实状态”。
  - 记录现在能跑什么、不能证明什么、下一步最该做什么。
- `HANDOFF.md`
  - 面向“下一位接手的人”。
  - 记录接手顺序、最短进入路径、容易踩坑的地方和本机验证结果。
- `docs/records/YYYY-MM-DD-*.md`
  - 面向“单次重要运行或阶段性记录”。
  - 记录具体一次实验/打通的上下文、命令、结果和解释。

### 原则三：把仓库当作未来要 release 的开源项目维护

这意味着：

- 仓库根目录只保留源码、配置、必要文档和轻量数据接口。
- conda 环境、editable 第三方 checkout、checkpoint、预测输出、loss 图等重产物都放到仓库外。
- 文档不能只写给当前操作者看，也要写给未来的开源用户和接手开发者看。
- 目录命名、文件命名、运行命令和路径约束都要可解释、可复现、可清理。

当前本机约定：

- Python 环境：`/root/miniconda3/envs/tooluse-llf`
- 重产物根目录：`/root/autodl-fs/tooluse-artifacts`

## 文档结构

文档按职责分层，不再把不同用途的内容混在一起。

### 根目录文档

- `README.md`
  - 面向仓库访问者和潜在开源用户。
  - 说明项目目标、快速开始、仓库结构和产物策略。
- `STATUS.md`
  - 当前真实状态。
- `HANDOFF.md`
  - 跨机器、跨会话交接。
- `2026-03-31-nips2026-function-calling-idea-draft-v2.md`
  - 当前论文主草稿。

### `docs/` 下的文档

- `docs/PROJECT_RULES.md`
  - 本文件，记录原则和维护规范。
- `docs/llamafactory-baseline.md`
  - baseline runtime 与评测使用说明。
- `docs/records/`
  - 日期化的实验记录、bring-up 记录、资源估计、研究状态。
- `docs/superpowers/plans/`
  - 执行计划和结构化分解。
- `docs/prompts/`
  - 仍有参考价值的提示词资产，但不是主线入口。
- `docs/archive/`
  - 已归档材料，默认不删除。

## 文档维护规则

### 写新文档前先判断放哪

- 如果它是仓库入口、项目总状态或交接说明，放根目录。
- 如果它是一次实验、一次 bring-up、一次审计，放 `docs/records/`。
- 如果它是执行计划，放 `docs/superpowers/plans/`。
- 如果它已经不影响主线决策，移到 `docs/archive/`。

### 不要再出现的情况

- 在根目录堆临时 scratch note。
- 把实验结论只写在终端输出里，不写回文档。
- 同一条关键原则在多个地方重复但长期不同步。
- 把本机重路径硬编码进多个文档却不统一维护。

### 建议的写法

- 运行记录优先写清楚：目的、非目标、命令、结果、解释、下一步。
- 判断性内容要和证据分开写，避免把工程打通说成论文结论。
- 面向开源发布的入口文档，避免只对内部上下文可读。

## 工程与文档之间的关系

文档不是附属品，而是 runtime、实验和论文推进的一部分。

- 新的运行路径打通后，要把命令和路径写回文档。
- 路径迁移后，要把旧路径引用清掉。
- 如果新增了比旧路径更成熟的方案，文档必须明确“谁是主线，谁是 legacy”。
- 如果代码还保留 legacy 路径，文档必须说明保留原因和删除条件。
- 如果历史计划文档仍提到已删除路径，以当前代码、`README.md`、`STATUS.md` 为准。

## 当前固定阅读路径

新接手的人默认按这个顺序读：

1. `README.md`
2. `docs/PROJECT_RULES.md`
3. `STATUS.md`
4. `HANDOFF.md`
5. 论文主草稿
6. 相关 runtime/实验记录

补充约定：

- 根目录 `STATUS.md` 只表示“当前总状态总览”。
- `docs/records/` 只表示“阶段记录、实验记录、资源记录”。
- 以后不再新建 `docs/status/` 这种容易和 `STATUS.md` 撞名的目录。
