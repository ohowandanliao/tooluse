# RULES

这个文件是当前仓库的用户规则入口。

目的只有两个：

- 把你明确提出的原则集中放到一处；
- 以后你要改规则时，优先只改这一个文件。

如果 `RULES.md`、`README.md`、`STATUS.md`、`HANDOFF.md`、`docs/PROJECT_RULES.md` 之间出现冲突，以 `RULES.md` 为准，然后再把其它文档同步修正。

## 怎么用

建议以后按下面的方法维护：

1. 你新增原则，直接加在本文件。
2. 你修改已有原则，直接改本文件原文。
3. 如果某条原则已经不适用，也直接在本文件删除或改写。
4. 每次原则变更后，同一轮工作里同步更新：
   - `STATUS.md`
   - `HANDOFF.md`
   - `docs/PROJECT_RULES.md`

## 当前原则

### 原则一：尽量不自己造轮子，但不是完全不自己造轮子

- baseline、训练 runtime、数据导出、评测、依赖安装，优先复用成熟开源方案。
- 只有在研究核心假设无法表达、现成工具不稳定、或者只差很薄的一层 glue code 时，才自己实现。
- 一旦成熟工具路径已经跑通，自写替代品就不再是主线。

当前落地解释：

- baseline runtime 优先走 `LLaMA-Factory`。
- 之前那些只产出 smoke stub 的手写 baseline trainer，不要再回来了。

### 原则二：必须随时更新 `STATUS.md` 和 `HANDOFF.md`

- 这不是“有空再补文档”，而是工作流程的一部分。
- 只要环境、路径、实验结果、判断、下一步建议发生变化，同一轮工作里就更新。

职责分工：

- `STATUS.md`
  - 记录当前仓库真实状态。
- `HANDOFF.md`
  - 记录下一位接手者最短接管路径。
- `docs/records/YYYY-MM-DD-*.md`
  - 记录单次重要实验、bring-up、审计过程。

### 原则三：把仓库当作未来要 release 的开源项目维护

- 仓库里只放源码、配置、必要文档和轻量接口。
- conda 环境、第三方 editable checkout、checkpoint、预测输出、图表等重产物都放仓库外。
- 文档不能只写给当前操作者看，也要让未来外部用户和接手者看得懂。

当前推荐约定：

- Python 环境名：`tooluse-llf`
- 重产物根目录：repo 外的 `ARTIFACT_ROOT`
- 推荐放法：仓库同级目录，例如 `../tooluse-artifacts`

### 原则四：不确定的东西先查外部做法，再决定是否推进

- 如果某个 benchmark 是否该当训练集、某种评测口径是否合理、某条训练配方是否常见，这类问题先查外部做法。
- 优先看：
  - 官方 benchmark 文档
  - 原论文
  - 官方仓库
  - 主流开源工程
  - 模型卡
- 如果外部做法没有共识，就不要把当前仓库里的探索路径包装成默认主线。

### 原则五：仓库内要保留可复现实验证据，但不提交权重

- 已经跑过的重要实验，至少保留：
  - 运行命令
  - `train_results.json`
  - `predict_results.json`
  - `toolcall_eval.json`
  - 必要训练日志
- 如果原始预测输出本身体量很小，也可以一起保留，作为“确实跑过”的证据。
- 权重、checkpoint、模型缓存和其它明显重产物，仍然不应提交到仓库。

当前落地解释：

- 这类证据统一放到 `results/`。
- `results/` 是给 release 仓库看的证据层，不是权重仓库。
- 当前 `pilot_v1` 的本机 run 已经开始按这条规则整理。

### 原则六：如果需要，可以整理 prompt 让用户去问 `GPT-5.4-pro` 做扩展或纠偏

- 当方案需要更强外部视角、更多备选路线、或者需要对当前判断做进一步 challenge 时，可以主动整理一份高质量 prompt。
- 这份 prompt 应该尽量包含：
  - 当前问题背景
  - 已有实现或实验现状
  - 明确想让对方扩展、纠偏或审查的点
  - 输出格式要求
- 但外部模型的回答不能直接替代本仓库里的验证、实现和文档更新。

当前落地解释：

- 可以把 `GPT-5.4-pro` 当高质量外部审阅者或方案扩展器。
- 不能把“问过更强模型”当成跳过本地核查和实验验证的理由。
- 如果真的用了这条路径，最好把采用了什么 prompt、采纳了什么判断，写回 `STATUS.md` 或对应记录文档。

### 原则七：做外部检索时，尽量优先更新的资料；旧资料要么够新，要么够硬

- 搜论文、工程、benchmark 用法、训练配方时，优先看 `2025` 年及以后的资料。
- 如果某个主题找不到足够新的高质量资料，才回退到更早的来源。
- 一旦回退到 `2025` 年以前，优先选择：
  - 影响力大的论文
  - 官方文档
  - 官方仓库
  - 社区里长期被引用的主流工程
- 不要为了凑引用，拿很旧但边缘、影响力弱、已经明显过时的材料当主要依据。

当前落地解释：

- 对研究方向、baseline 选择、benchmark 定位、训练 recipe 判断，默认先搜近两年的资料。
- 如果旧资料仍被保留在判断链里，要么它是该方向的源头工作，要么它仍然是当前社区的高影响力参考。
- 文档里如果引用了较旧材料，最好顺手说明它为什么仍值得保留。

## 当前补充约束

这些也是你已经明确表达过、应该长期保留的约束：

### 环境与路径

- 不要在仓库目录里单独再放一套 conda。
- 优先使用机器已有的 conda 根目录。
- pip 安装如果慢，可以用国内源。
- 代码拿到新机器后，应该能在没有 Codex 的情况下靠仓库文档快速恢复环境并开始实验。
- 所以依赖版本、第三方 runtime commit、验证命令和路径约定，必须写进仓库。
- 平台镜像/系统盘快照可以默认保留，但 repo 内仍然必须保留裸 Linux 的恢复路径。
- 不要把某个固定 CUDA 版本写成所有机器唯一可用的前提。

### 文档与语言

- `STATUS.md` 和 `HANDOFF.md` 用中文写。
- 文档尽量收束，避免同类信息在多个地方长期分叉。

### 仓库清洁

- 不该提交的东西，要么删掉，要么 ignore。
- 环境、缓存、训练产物、外部源码 checkout 不要混进仓库提交。
- 但轻量实验结果证据不属于“脏东西”，不要和权重一起一刀切排除。

### 研究推进方式

- 先推进真实可用的 baseline 和数据链路，再谈更复杂的方法侧实现。
- benchmark、训练集、评测集的角色要分清楚，不要混着用。
